Data storage

For this prototype, records are saved in browser localStorage using lower snake-case table-like keys. The included file data/schema.sql shows the matching ERD-style SQL table design.

SQL tables used: League, Team, Player, CoachUser, TeamCoach, Game, GameRoster, PlayerGameStatus, Foul, Injury, Substitution, AIQuery, and Report. LocalStorage uses matching lower snake-case keys such as lgmis_players and lgmis_player_game_statuses.

User steps

How a coach or admin should use the prototype during the demonstration.

Step 1: Login with a coach or admin demo account.
Step 2: Coach adds a team and jersey-only roster using Team Add.
Step 3: Coach manages jersey status in Team Manage using the color-coded indicators.
Step 4: Coach creates a game record and enters game stats on Game Tracker.
Step 5: Coach or admin views the player and game reports. SQL buttons stay as placeholders until database integration.
Step 6: Admin logs in to add or delete adult coach/admin users and assign users to teams.

Interface standards and design decisions

These standards keep the site consistent, easy to grade, and usable by non-technical coaches.

Visual standards

  • Maryland theme using red, black, gold, and white.
  • Large buttons, rounded cards, and high contrast text for game-day use.
  • Consistent top navigation across all screens.
  • Responsive layout for laptop, tablet, and phone demonstrations.

Status standards

  • Red: jersey record needs time or needs monitoring.
  • Green: jersey record is eligible or has met the playtime requirement.
  • Blue: jersey record has an injury status.
  • Gray: jersey record is unavailable, inactive, or disqualified.

Data standards

  • Players are identified by jersey number only.
  • Player names, initials, and other minor-identifying information are not collected.
  • Jersey number is unique within a team, but can repeat across different teams.
  • Forms use field names that match the SQL schema.
  • Future SQL connection should replace localStorage, not the UI layout.

Security and privacy standards

  • Role-based screens for coach, assistant coach, and league admin.
  • Adult coach/admin accounts can use names; roster players cannot.
  • Future backend should use encrypted transmission and hashed passwords.
  • Admin-only user management.
